# Post-Mortem: Why OnGo's Agents Hallucinated

**Date:** 2026-06-26
**Author:** Claude Code (investigation requested by founder)
**Status of system:** Autonomous building **paused**. No fix deployed. This is
diagnosis, not remediation.

---

## 1. Summary

OnGo's autonomy engine produced two visible failures:

1. **Generic ideas** — the Research agent surfaced opportunities like
   "AI-powered automation" / "autonomous system" instead of specific, buildable
   products.
2. **Incoherent repos** — incubated projects were filled, file by file, with
   code that didn't fit together: hallucinated imports, invented APIs,
   duplicated or contradictory types, placeholder logic.

Neither is a bug in the *platform*. The Brain, approval policy, audit log, and
data model all worked as designed. The failure is **the executor and the loop
around it**: under-powered free models were pushed to do open-ended,
multi-file software engineering with **no verification, no memory, and no human
review of the creative output**, at high frequency. That combination guarantees
confident-but-wrong output — and because it commits to real public-facing
repos, it damages the exact thing OnGo exists to build (a credible GitHub
profile and portfolio).

---

## 2. What "hallucination" actually meant here

Two distinct mechanisms, often conflated:

- **Idea-level hallucination** (Research agent): with thin or no grounding, a
  language model defaults to the statistical center of "startup idea" — i.e.
  the most generic possible phrasing. It isn't lying; it has nothing specific to
  anchor on, so it regresses to vague.

- **Code-level hallucination** (Developer agent): asked to write the "next file"
  of a project it cannot fully see, a model invents the pieces it's missing —
  module names, function signatures, library APIs — because producing *something
  plausible* is what it's trained to do. Across many cycles with no shared
  memory, these inventions drift apart and the repo stops cohering.

The root causes below explain why both were not just possible but **likely**.

## 3. Root causes (ranked by impact)

### RC-1 — Weak free models doing a job they can't do reliably
The chain in production was free/low-tier models: Groq `gpt-oss-120b`,
`qwen3-32b`, `llama-3.3-70b`, plus OpenRouter free tier and `cohere/north-mini`.
These are fine for short, well-scoped, grounded responses. They are **not**
reliable for autonomous, open-ended, multi-file software construction — the
hardest thing you can ask an LLM to do. They lose track of cross-file state,
invent APIs, and can't self-correct. **No prompt fixes a capability ceiling.**

### RC-2 — No verification gate on generated code (the biggest structural gap)
In `runProjectDevelopment` (`autonomy.service.ts`), whatever the model returns
is committed after only two checks: a **path-safety** filter (`sanitizeDevPath`)
and a **length** check (`content.length < 20`). There is:

- ❌ no `tsc` / type-check
- ❌ no build
- ❌ no lint
- ❌ no test run
- ❌ no "does this import resolve?" check

`code.generate` is classified **SUGGESTED** in the approval policy → it
**executes immediately** and merely logs a flag. So broken code goes straight
into the repo. The platform's safety work went into *path traversal / CI
injection / secrets* (real and good) — but **none** of it checks whether the
code is *correct*. A hallucinated `import { core } from "./core"` against a
`core.ts` that defines something else entirely passes every gate.

### RC-3 — The open-ended phase is blind to file *contents*
After the 6-file scaffold (`DEV_PLAN`), the loop enters an open-ended phase
where the agent picks "the next most valuable file." But it is given only
`github.listRepoFiles(repo)` — **a list of file paths, not their contents.**
So the agent writes new code that imports from and references files **it cannot
read.** It is structurally forced to guess their exports and signatures. This is
a near-guaranteed source of drift and duplication, independent of model quality.

### RC-4 — No project-level memory or context
Every cycle is an independent `brain.dispatch`. The agent's only context is a
one-line `project.description`. There is no persistent record of decisions,
naming conventions, the tech approach, or what was built last cycle. With no
memory, the model re-derives the project from scratch each time — which is both
why ideas stay generic and why files contradict each other.

### RC-5 — Volume optimized over quality
The interval was pushed to 5 minutes and the incubator + dev loop run every
tick. This maximizes *commit frequency*. But un-reviewed, unverified output at
high frequency just produces **more junk faster**. A green contribution graph
made of incoherent AI commits is worse than empty for the stated goal — anyone
evaluating the profile (a client, a recruiter) sees low-quality autogenerated
slop, which is a negative signal, not a positive one.

### RC-6 — Fallbacks masked failure
`DEV_PLAN` steps each have a `fallback` that "guarantees a real, sensible
commit even if the model output is unusable." Operationally this means a failed
generation **looks identical to a successful one** — a commit still lands. There
was no signal distinguishing "the model did good work" from "the model failed
and we shipped a stub." Failures were invisible, so they accumulated unnoticed.

### RC-7 — The creative/quality-critical actions were tuned to auto-execute
OnGo's human-in-the-loop design is genuine, but the actions that most needed a
human — `github.repo.create` and `code.generate` — are **SUGGESTED**
(execute-then-log), not **MANDATORY** (block-until-approved). So the founder
never reviewed a repo or a code change *before* it became public-facing history.
The policy was tuned for throughput over a quality checkpoint exactly where a
checkpoint mattered most.

### RC-8 — Grounding fix existed but was never live (and wouldn't have been enough)
A rework to ground the Research agent (real HN + GitHub-trending signals via
`fetchSignals`, plus a specificity-forcing prompt in `buildResearchPrompt` that
bans generic phrasing) was written but **never built or deployed**. So the
generic ideas you saw were from the *un-grounded* version. Worth noting: even
with grounding, RC-1 (weak models) means specificity would have improved but
coherence of the *built* projects would not.

## 4. Why the architecture is NOT to blame

It's important to be precise so the fix targets the right layer:

- ✅ **OnGo Brain** correctly gates every action and writes an audit row.
- ✅ **Approval policy** is a clean, single source of truth (deny-by-default for
  unknown actions).
- ✅ **Path safety** (`sanitizeDevPath`) genuinely prevents traversal, dotfile,
  `.github/workflows` CI-injection, and secret-file writes.
- ✅ **Data model** (opportunities → projects → dev steps) is coherent.
- ✅ **Showcase/profile sync** is read-only and safe.

The engine did *exactly what it was told*. It was told to let weak models write
software unsupervised and commit it. That instruction was the defect.

---

## 5. The core lesson

> **Autonomy without verification is just automated mistake-making.**

An autonomous software factory needs the same things a human team needs to ship
quality: a clear spec, memory of decisions, and — above all — a **gate that runs
the code and rejects it when it's wrong** before it reaches anything public.
OnGo built the org chart and the audit trail. It skipped the CI gate and the
senior reviewer. Those two are the difference between "impressive demo that
embarrasses you in your repo history" and "a portfolio you'd show a client."
